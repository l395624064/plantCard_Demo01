- 仅当前项目生效（project skill）。

## 架构总原则

- 先保证模块单一职责，再考虑分层形式。
- 功能模块优先采用 MVC 思路，但不是所有模块都必须有 model/view。
- 基础设施模块（例如 `{{infra_module_alias}}/*`）可以只保留必要的基础设施代码。
- 禁止为了“结构完整”而创建无业务价值的空层。

## 游戏流程独立模块规则（硬规则）

- 游戏规则、流程推进、胜负判定必须放入独立流程模块。
- 模块命名允许同义（例如 `{{flow_module_alias}}/*`、`gameFlow/*`），但同一项目内必须统一一种命名风格。
- 禁止将流程核心逻辑散落在入口文件、视图文件或临时工具文件中。

## 编排目录职责规则（硬规则）

- `{{orchestration_module}}/*` 仅允许承载跨模块编排/协调逻辑。
- `{{orchestration_module}}/*` 可做跨模块调用编排，但不得长期承载具体领域核心实现。
- 领域核心实现（例如棋盘/卡牌/地块/植物）必须归属各自独立模块。

## 禁止冗余状态构建层（硬规则）

- 禁止新增仅做字段拼装/透传且无独立业务价值的状态构建器文件。
- 这类层默认不应存在。
- 该类逻辑应：
  - 直接放在入口协调层，或
  - 放在真实流程/领域模块（当其职责确实属于该模块）。

## 适配归属规则（硬规则）

- 屏幕适配、Canvas 适配、安全区与缩放计算逻辑，统一放在适配模块（例如 `{{adaptive_module_root}}/*`）。
- 禁止将适配逻辑放在业务模块目录中。

## 全局管理器（缺失时才创建）

全局路径由项目配置层定义：

- `{{ui_manager_path}}`
- `{{event_manager_path}}`
- `{{model_manager_path}}`

最小接口要求：

- `UIManager`
  - `open(uiKey, data?)`
  - `close(uiKey?)`
  - `closeAll()`
  - `curui`（最后打开的 UI 实例）
- `EventManager`
  - `on(event, handler, target?)`
  - `off(event, handler?, target?)`
  - `emit(event, payload?)`
  - `once(event, handler, target?)`
- `ModelManager`
  - `register(type, ctor)`
  - `enable(type)`
  - `get(type)`
  - `disable(type)`
  - `destroy(type)`

Model 使用要求：

- 模块 model 类型需先在 `ModelManager` 注册；
- 模块功能使用前需先启用对应 model 数据源/实例。

## 模块目录规范

以 `feature` 模块为例：

- `{{module_root}}/feature/FeatureManager.<ext>`
- `{{module_root}}/feature/model/FeatureModelBase.<ext>`
- `{{module_root}}/feature/model/FeatureModelBuilder.<ext>`
- `{{module_root}}/feature/view/ui/*`
- `{{module_root}}/feature/view/item/*`
- `{{module_root}}/feature/view/panel/*`
- `{{module_root}}/feature/utils/*`
- `{{module_root}}/feature/FeatureEnum.<ext>`

其他模块按模块名等价替换。

- 说明：
  - 以上是“按需推荐结构”，不是要求每个模块都必须包含全部层级。
  - 不允许仅为凑结构而新增空的 model/view 层。

## 命名规范（Enum/Type/Interface）

在 `<ModuleName>Enum.<ext>` 中：

- `type_<module>_xxx`
- `enum_<module>_xxx`
- `interface_<module>_xxx`

说明：

- `<module>` 必须小写（如 `card`、`parcel`、`audio`）；
- 有意义字段采用驼峰式命名。

## 正式模式类型安全规则（硬规则）

- 正式模式（非 MVP 模式）默认避免使用 `any`。
- `any` 仅在 `mvp_*` 模块中可不受限制使用。
- 正式模式下，跨文件调用应尽量避免传完整 `this`。
- 跨文件调用优先使用最小能力接口（例如 `interface_xxx_port`）。

## 对外暴露规则（硬规则）

- 普通模块对外默认通过 `<ModuleName>Manager` 单例暴露能力。
- 除全局/通用项外，尽量收敛导出面：
  - enum/type/interface/const/utils/通用方法。
- 禁止大量无业务价值的薄转发导出函数。

## Manager 体量与子模块规则（硬规则）

- MVP 模式下，manager 体量不做限制。
- 正式模式下：
  - 主 manager 代码量较小时可内聚完整业务（软阈值约 300 行），
  - 代码量增大或职责混杂时，必须拆分子 manager。
- 调用方式统一为：
  - `mainManager.subManager.method()`
- 主 manager 持有的子 manager 单例引用应为 `readonly`。

## Controller 目录规则（硬规则）

- 正式模式默认不创建/不使用 `controller` 目录。
- 输入编排逻辑应放在 view 或 view 子模块中，便于查阅与定位。
- 流程转发优先：
  - 单向通知使用 `eventManager.emit(...)`，
  - 需要返回值或同步决策时，直接调用 manager（或子 manager）。
- 禁止新增仅做转发且无独立价值的 controller 文件。
- MVP 模式可临时放宽，转正时必须回收并对齐正式规则。

## Service 文件规则（硬规则）

- 默认不允许创建独立 `*Service.<ext>` 文件或 `service/*` 目录。
- 优先使用 manager 子模块化。
- 允许 `serviceManager` 作为 manager 下的子域命名（例如 `xxxManager.serviceManager`）。

## Manager 调试挂载安全规则

- 若需要把 manager 单例挂到 `window` 便于调试，必须先做运行环境判断：
  - 使用 `if (typeof window !== 'undefined') { ... }`
  - 再执行 `window['模块名Manager'] = 模块Manager`

## View 与 Manager 约束

- `view/*` 建议约 300 行内（软限制）。
- 若 UI 内部组件复杂，优先拆分到 `item` 或 `panel`。
- 视图实现必须保持清洁：
  - 清理未调用方法、未使用字段、未使用导入；
  - 视图入口类仅保留门面职责；
  - 交互/绘制/几何等重逻辑下沉到专门模块。
- `Manager` 必须保持单一职责与唯一性：
  - 例如 `AudioManager` 只管理音频，不混入其他模块逻辑。

## 类型文件单一职责规则（硬规则）

- 禁止“全能类型文件”混放多个领域的类型与职责。
- 类型应按领域拆分。
- 跨领域共享类型应保持最小且稳定。
- 允许短期过渡导出文件，但需按计划逐步移除。

## 防碎片化拆分规则（硬规则）

- 禁止对直接依赖、强上下文耦合的小体量代码进行过度拆分，避免产生过多碎片文件。
- 对于直接依赖代码，若实现体量低于约 60 行，默认保持在当前归属文件中。
- 目标是在“功能单一职责”和“人工审阅可读性”之间取得平衡。
- 建议拆分时机：
  - 代码扩写并持续超过约 60 行，
  - 已出现明显多职责混杂，
  - 出现稳定跨文件复用需求。
- 参考阈值：
  - 小于 60 行：默认不拆；
  - 60-100 行：结合复杂度和耦合度评估；
  - 大于 100 行：优先拆分。
- 允许提前拆分的例外：
  - 安全或关键生命周期边界；
  - 明确的公共复用能力；
  - 可测试性/隔离性需要；
  - 用户明确要求拆分。

## MVP 模式规则（硬规则）

- 当用户明确提出以下指令时，必须按 MVP 生命周期默认规则执行，无需额外提示词模板：
  - `xx模块需要使用mvp模式开发`
  - `xx模块需要从mvp模式转正`
- 可选第三条指令：
  - `xx模块的mvp方案废弃`

### MVP 开发默认行为

- 功能代码放在：
  - `{{mvp_root}}/*`
- 必须满足三条约束：
  1. 单入口接入（`Mvp<Feature>Entry.<ext>` 作为唯一接入入口）；
  2. 明确开关控制（关闭后主链路行为不变）；
  3. 不污染核心契约（MVP 阶段默认不修改 `{{infra_module_alias}}/*`、`{{flow_module_alias}}/*` 的共享契约，除非用户明确要求）。

### MVP 转正默认行为

- 将确认保留的 MVP 逻辑迁移到正式模块，并遵循当前 code-style-skill 的边界与命名规则。
- 清理 MVP 阶段临时耦合代码。
- 转正后删除 `{{mvp_root}}` 目录与残留接入点。

### MVP 废弃默认行为

- 关闭开关、移除接入入口、删除 `{{mvp_root}}`。
- 确保删除后主链路行为不受影响。

## 契约模式规则（硬规则）

- 契约模式是可选开发模式，默认关闭。
- 核心目标：
  - 降低“需求语义与运行时行为”风险，
  - 将用户意图先固化为可执行契约，再进入实现。
- 触发方式：
  - 用户明确要求使用契约模式，或
  - 助手识别需求歧义较高时，先询问是否开启契约模式。
- 退出方式：
  - 用户可随时退出契约模式并切回默认模式；
  - 切换前助手必须先总结已确认契约事实，避免信息丢失。

### 契约模式工作流程

1. 契约收集阶段（不写代码）。
2. 契约确认阶段（用户确认条款）。
3. 实现阶段（严格按已确认契约开发）。
4. 回归阶段（按可执行回归条目验证）。

- 契约确认完成前，助手不得进入实现阶段。

### 契约输入方式

- 填空模板模式（新手友好）：
  - 助手提供模板，
  - 每个字段都要给简短示例说明，
  - 用户需要时，助手应提供一份简单示例填写稿用于参照。
- 一问一答模式（助手主导）：
  - 每轮最多提 1-3 个核心问题，
  - 每轮必须告知预计剩余轮次，
  - 若回答对实现仍不充分或存在歧义，必须继续追问。

### 契约子模式用户简述（强制）

- 契约模式开启时，助手必须先给用户提供“精简双模式说明”，再开始契约收集：
  - **模板模式**
    - 定位（主导方）：用户主导结构化填写；
    - 交互方式（节奏）：轮次较少、单轮信息量较大；
    - 适用人群（场景）：需求较清晰、倾向直接填写模板的用户。
  - **问答模式**
    - 定位（主导方）：助手主导问题树；
    - 交互方式（节奏）：每轮 1-3 问，并告知预计剩余轮次；
    - 适用人群（场景）：新手用户、需求歧义高或交互复杂场景。
- 首轮说明需保持精简，避免一次性输出过多模式细节。

### 契约合规规则

- 若用户回答不符合契约模式要求：
  - 助手必须提醒按规范补充回答，或
  - 询问是否退出契约模式。
- 契约模式下，助手不得在信息不足时默认猜测并直接实现。

### 可执行回归条目规则

- 契约模式输出必须包含可执行回归条目。
- 每条回归条目必须包含：
  - 明确的用户操作步骤，
  - 明确的可观察预期结果。
- 禁止只写抽象描述（例如“应该没问题”）。
- 建议至少 3 条；交互复杂功能建议 5 条以上。

### 模式选择告知规则

- 用户提出新功能但未明确模式时，助手必须先询问模式选择。
- 助手需简要说明每种模式的：
  - 适用用户类型，
  - 主要收益，
  - 用户需要提供的信息。
- 当前模式：
  - 默认模式（速度优先），
  - MVP 模式（试验优先），
  - 契约模式（语义稳定优先）。

### 模式元信息规则

- 后续每新增一种开发模式，必须同步补充：
  - 适用用户类型，
  - 适用场景，
  - 用户输入要求，
  - 成本/收益说明。

## 辅助引导规则（硬规则）

- 新增“辅助引导”能力组，默认关闭，用户可显式开启。
- 当前辅助引导包含：
  - 任务启动卡，
  - 模式切换守卫，
  - 交付影响卡，
  - tmp 流程日志引导。

### 任务启动卡

- 触发时机：
  - 接收到新的功能/任务请求，
  - 当前会话内需求范围发生明显变化，
  - 开发模式发生切换。
- 助手必须先完成需求对齐，再进入编码；若有缺失项，必须明确告知缺失内容。
- “实现前完成条件”与“任务启动卡”合并为同一步骤，不重复执行。

### 模式切换守卫

- 用户提出新功能但未声明开发模式时，助手必须先询问模式选择。
- 助手需简述：
  - 模式目标，
  - 用户投入成本，
  - 主要收益。

### 交付影响卡

- 实现完成后，助手应输出：
  - 变更涉及模块/文件范围，
  - 主要行为风险点，
  - 建议回归项，
  - 轻量质量评分。
- 契约模式可省略评分细节（因其契约校验本身已较严格）。

### tmp 流程日志引导

- `tmp` 流程日志默认关闭。
- 用户可主动开启 `tmp` 流程日志。
- 当用户开启“辅助引导”时，必须同步开启 `tmp` 流程日志。

## 协作分层与闭环执行规则（硬规则）

- 该规则适用于所有开发模式（默认 / MVP / 契约）。
- 用户协作分层：
  - 无任何开发经验，
  - 具备一定开发经验，
  - 有充足开发经验。
- 分层来源：
  - 助手可基于需求文档质量与术语使用给出建议层级，
  - 最终层级以用户确认结果为准。
- 首次使用可询问用户层级；用户可随时切换层级并立即生效。
- 若用户未声明层级，默认层级为：**具备一定开发经验**。

### 闭环执行策略

- 闭环执行指：需求对齐后，助手连续实现，避免频繁中断提问。
- 所有层级下，运行时正确性是不可破坏底线。
- 禁止为了满足风格规则而牺牲运行时正确性。

### 分层执行策略

- 无任何开发经验：
  - 不启用闭环执行，
  - 允许开发中提问，且尽量使用直白话术。
- 具备一定开发经验：
  - 启用闭环执行，
  - 默认不启用中断白名单，
  - 决策优先级：运行正确 > 功能完整 > 其他细节。
- 有充足开发经验：
  - 先完成严格需求对齐（无关键歧义），
  - 启用闭环执行，
  - 启用中断白名单，
  - 决策优先级：在不破坏运行时正确前提下优先遵循 code-style 规范。

### 中断白名单（启用时生效）

- 仅以下情况可中断提问：
  - 不可逆/高风险操作，
  - 缺少必须外部输入（账号/权限/密钥等），
  - 关键需求冲突且会改变核心行为。
- 非白名单问题应后置到交付说明中，作为可选跟进项。

### 非阻塞问题后置规则

- 在实现过程中，非阻塞问题不应中断开发流程。
- 助手应继续执行，并在交付后统一后置说明。
- 后置项统一归类为：
  - 可选优化项，或
  - 不影响当前行为闭环的待确认项。

### 层级切换回显（建议）

- 层级切换后，助手应简要回显：
  - 当前是否启用闭环，
  - 当前是否启用中断白名单，
  - 当前决策优先级。

### 与契约模式关系

- 契约模式负责“实现前需求固化”。
- 协作分层与闭环规则负责“实现阶段中断行为”。

## 迁移模式规则（硬规则）

- 新增“迁移模式”，用于将旧代码按当前 skill 规则对齐。
- 用户最小输入仅需：
  - 目标文件路径或目录路径。
- 助手应基于当前 code-style 规则自主执行迁移。
- 默认采用保守迁移策略：
  - 不主动改变运行行为，
  - 优先进行结构/可读性对齐，
  - 必要时分步迁移降低风险。
- 若迁移中识别到高运行风险改动，助手必须暂停并向用户确认后再继续。

### 迁移模式验收规则

- 迁移完成后，助手必须提供迁移验收说明，至少包含：
  - 结构变化摘要，
  - 运行时行为不变性说明，
  - 高风险点（如有），
  - 建议回归检查项。

## 工具函数放置规范

- 高复用、跨模块工具放全局工具（如 `TweenUtils.<ext>`、`ArrayUtils.<ext>`、`ColorUtils.<ext>`）。
- 仅模块内部使用的工具放 `<module>/utils/*`。
- 模块内 utils 允许依赖同模块 manager/model，以减少无意义碎片化。
- 对外尽量保持单入口调用：
  - 外部只调用一个公开方法，
  - 内部可拆分私有步骤，但不向外暴露实现细节。

## 常量放置规范

- 全局共享常量统一放在：
  - `{{global_const_file}}`
- 模块内常量放在模块目录下，命名为：
  - `<ModuleName>Const.<ext>`
  - 例如：`{{module_root}}/card/CardConst.<ext>`

## 模块临时代码规则

- 某个模块的测试性/实验性代码，统一放在：
  - `{{module_tmp_rule}}`
- `<module>/tmp/*` 下代码视为可抛弃代码，可随时删除。
- 临时代码不允许成为稳定功能模块的运行时依赖。
- 核心模块逻辑不得依赖 `<module>/tmp/*` 中的文件。
- 用户若明确指定删除 `tmp/` 目录下的文件或子目录，可立即执行清理。
- 助手自动清理时，`{{trace_runtime_dir}}/*` 不适用“可随时删除”，必须遵循下文 Tmp 流程日志清理规则。

## Tmp 流程日志规则（硬规则）

- 采用 tmp 流程日志方案时，标准 trace 调用文件统一放在：
  - `{{trace_runtime_dir}}/*`
- 助手必须给用户两个选项：
  - 用户指定 `<tmp 流程日志>` 工具类目录；
  - 创建 `<tmp 流程日志>` 工具类（用户可指定目录；若未指定，则由助手默认创建在 `{{trace_runtime_dir}}/*`）。
- 若用户未指定目录且项目中已存在可用 tmpTrace 实现，助手必须优先复用现有目录与实现，避免重复创建。
- “可用 tmpTrace 实现”判定至少满足：
  - 存在 `TmpTraceEnum.<ext>`（或等价命名）；
  - 存在 `TmpTraceManager.<ext>`（或等价命名）；
  - manager 具备最小接口能力：存档、更新、删除。
- 若用户指定目录不存在，助手应自动创建目录，并在交付信息中回显最终创建路径。
- 运行时 trace 输出文件默认归口在：
  - `{{trace_runtime_dir}}/*.jsonl`
- 在用户明确验收通过前，不得删除该功能对应流程日志文件。
- 仅以下场景可豁免创建：
  - 纯文案替换，
  - 纯静态资源替换，
  - 无逻辑变更。

### 工具类最小结构规则

- `tmp` 流程日志工具类至少包含以下两个核心文件（或等价命名文件）：
  - `TmpTraceEnum.<ext>`：定义日志数据结构，且字段模型必须遵循“AI理解优先原则”；
  - `TmpTraceManager.<ext>`：定义日志接口（至少覆盖存档、更新、删除）。
- `<ext>` 必须遵循当前项目主语言与文件格式约定（例如 `ts`、`js`、`cs`、`lua`）。
- trace 调用文件命名建议：
  - `tmpTrace_<FlowName>__<EntryFile>.<ext>`
- `unitId` 命名建议：
  - `<FlowName>_<EntryFile>_Uxx`
- `stepId` 命名建议：
  - `<FlowName>_<EntryFile>_Uxx_<Step>`

### 调用文件职责边界规则

- `tmpTrace_<...>.<ext>` 仅负责：
  - 打包 trace 数据，
  - 调用 `TmpTraceManager` 投递。
- 禁止在该文件内承载业务判断（状态判断、去重判断、流程分支）。

### 运行时写入规则

- 一旦调用 `TmpTraceManager`，默认执行：
  - 写入 trace 记录，
  - 控制台输出 trace。
- 写入策略：
  - 状态变化驱动为主，
  - 高频流程以 `100ms` 采样兜底，
  - 禁止逐帧无差别写入。
- 同一 `unitId` 的连续重复日志应做去重或降频。
- 单日志文件大小上限 `5MB`，超限后必须滚动分片。

### 数据结构规则（AI 可理解）

- “AI理解优先原则”仅用于 `<tmp 流程日志>` 数据结构定义，优先保证 AI 可还原执行链、定位异常因果、对齐验收清单。
- 数据结构可参考项目内现有 trace schema（如 `{{trace_schema_ref}}` 的语义），并允许按项目实际做等价扩展。
- “AI理解优先原则”最小验收标准：
  - 能还原执行链（步骤顺序与上下游关系）；
  - 能进行预期/观察对比；
  - 能定位异常因果（关键决策与原因可追踪）。
- 流程日志必须使用结构化字段，最小字段集：
  - `sessionId`
  - `feature`
  - `unitId`
  - `seq`（全局递增）
  - `phase`（`enter | state | exit | error`）
  - `action`
  - `effect`
  - `status`（`ok | fail`）
  - `ts`
  - `payload`（可选，限长）
- 一条日志只表达一个核心事件，不得混合多个动作与结果。

### 最小单元切分规则

- 流程日志最小单元默认按“完成清单项”切分（`unitId`）。
- 每个 `unitId` 必须同时具备：
  - 明确输入动作，
  - 明确状态变化，
  - 明确可观测结果。
- 若某 `unitId` 失败或歧义，允许在该单元内部按二分法加密埋点；
- 禁止全局无差别增加日志密度。

### 清理规则

- 用户手动清理：
  - 用户明确指定 `{{trace_runtime_dir}}/*` 下的文件或目录时，可立即执行清理。
- 助手自动清理仅在以下条件全部满足时允许执行：
  - 用户明确确认“功能验收通过”，
  - 对应最小回归集通过，
  - 完成交付归档（保留关键节点与清单映射），
  - 无阻塞级遗留问题。
- 助手自动清理采用静默保留期机制，默认保留 `1 个迭代` 后删除。

### 浏览器联调引导规则

- 首次安装 skill 或首次功能联调时，助手必须判断项目是否运行在浏览器环境。
- 若为浏览器环境，助手应直接执行 `{{browser_mcp_server}}` 接入流程并完成首次连通性自检，无需二次确认。
- 若 `{{browser_mcp_server}}` 接入或连通性自检失败，助手必须自动回退到 `tmp/*` 日志文件链路，并继续执行当前任务。
- 若非浏览器环境，默认使用 `tmp/*` 日志文件链路，不阻塞功能开发。

### 完成清单映射规则

- 每个新功能必须输出“完成清单”。
- 清单每一项必须绑定唯一 `unitId`。
- 流程日志必须覆盖全部 `unitId`。
- 验收结论必须基于“双证据”：
  - 清单项状态，
  - 对应 `unitId` 的日志证据。

## 工具链优先级总则（强制）

- 工具链选择与落地顺序固定为：
  1. 优先复用全网较成熟（或使用人数最多）的 MCP / skill / 工具；
  2. 若第 1 步不可用，则在 GitHub 检索 star 较高的 MCP / skill / 工具；
  3. 若第 1、2 步仍不可用，则复用项目内已有工具；
  4. 选定方案后，助手必须自动接通并完成基础调试验证；
  5. 若仍缺失，才允许新建工具实现，语言优先级：`Node.js > Python`。

### 外部工具准入门槛

- 外部方案“可用”至少满足：
  - 可安装，
  - 可运行，
  - 许可证可用于当前项目，
  - 与当前环境兼容。
- 若不满足，助手必须说明淘汰原因，再进入下一优先级。

### 自动执行与回退

- 命中可用工具链后，助手默认自动执行接通与最小调试，不以“建议用户手动处理”为主路径。
- 浏览器场景下，`{{browser_mcp_server}}` 连通后助手应自动完成链路检查与日志链路初始化。
- 自动执行失败时，助手必须自动回退到备用链路并继续任务（例如回退到 `tmp/*` 本地 trace 链路）。

### 运行与成本约束

- 高频日志必须采样/节流（如 `100ms`）。
- 同一 `unitId` 连续重复日志应去重或降频。
- 仅本地采集/终端打印不视为模型 token 消耗；
- 只有 AI 读取并分析日志内容时，才计入 token 消耗。

### 交付闭环与回执

- 工具链交付必须包含：
  - 最终采用方案（对应优先级层级），
  - 淘汰方案与原因，
  - 接通结果与基础调试结果，
  - 失败回退路径（如触发），
  - 启动/停用方式（可复现）。

## 工具链通用化规则（强制）

- `code-style-skill` 的工具链规则必须跨引擎通用，不绑定单一引擎、单一 MCP 名称或固定脚本实现。
- 规则正文仅描述“执行链路与验收口径”，不写死项目实现细节。
- 工具链执行必须遵循环境自适应流程：
  1. 环境识别（运行态与技术栈），
  2. 能力探测（MCP / skill / 项目内工具 / 日志出口），
  3. 按优先级选型并自动接通验证，
  4. 失败自动回退。
- 正文禁止写死具体项目工具名或路径；具体值应通过占位符表达。
- 具体项目路径、工具名、实现方式只能放在 `[项目示例 - 非强制]` 中。

## 事件系统规范

- 事件定义保持单文件集中管理，不按模块拆分事件定义文件。
- 事件定义路径统一：`{{event_definition_file}}`。
- 事件命名使用下划线风格：
  - 模块事件：`card_xxx`、`parcel_xxx`、`audio_xxx`
  - 公共事件：`app_xxx` 或 `global_xxx`
- 例外：专门的全局事件定义文件可使用较长、语义完整的事件命名（项目示例见项目配置层定义）。
- `EventManager` 使用全局单例方式（不引入 Sender 包装层）。
- Manager 生命周期事件处理约束：
  - 在 `addEvents()` 注册
  - 在 `offEvents()` 反注册
  - `dispose()` 必须调用 `offEvents()`

## Git 管理约束（当前项目）

- 编码/提交前先检查 `git status` 与相关 `git diff`。
- 提交按任务粒度聚焦，避免混入无关改动。
- 未经用户明确要求，禁止破坏性操作：
  - `reset --hard`
  - 强制推送共享分支
- 未经用户明确要求，不改写历史（如 amend）。
- 若仓库有无关未提交改动，必须保留，禁止回滚用户改动。
- 当前 skill 不强制分支命名与 PR 流程。

## 执行流程

编码前：

1. 检查全局 manager 是否已存在。
2. 优先复用现有基建。
3. 缺失时仅补最小可用 manager。
4. 按模块规范创建目录和骨架。
5. view 按 ui/item/panel 控制复杂度。
6. 工具函数放置到正确层级。
7. 检查 git 状态，避免误改无关文件。

编码后：

1. 校验命名与目录规范。
2. 校验 model 注册/启用流程。
3. 校验 view 复杂度与拆分合理性。
4. 复查 git diff，仅保留目标改动。
5. 复查是否产生了不必要的小碎片文件（直接依赖且小于 60 行的代码默认不拆）。

## 歧义处理（必须先问用户）

遇到以下不明确项，必须先沟通再执行：

- 全局 manager 的确切路径或组织方式是否变更；
- ui/event/model 需要超出最小接口时的签名；
- view 300 行阈值在当前任务是否有例外；
- `<ModuleName>Enum.<ext>` 命名前缀是否有模块级例外；
- 是否允许 manager 做临时跨模块编排。

- 最终整合触发方式：
  - 用户明确提出“最终整合请求”，或
  - 助手在必要时主动询问“是否进入整合最终版本”并获用户同意。
- 进入最终整合阶段后，默认冻结新增条目；
